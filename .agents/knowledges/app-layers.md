## Main Layers

### 1. Presentation Layer
The Presentation Layer is the entry point of the application. It is responsible for handling incoming requests and returning responses to the client.
- Request validation must be performed exclusively within this layer.
- This layer can only interact with functions exposed by the [Usecase](#2-usecase-layer-service-layer) layer.
- All request and response structures must reside in the [Entity](#1-entity) layer, even if they contain only a single field.

#### 1.1. API Layer (Server REST)
The API Layer is a specific type of Presentation Layer responsible for handling REST API requests.

It is common to organize the API layer into different scopes, such as:
- Public API (accessible to external clients)
- Service API (accessible only by internal microservices)

*Note: Tracing and context propagation are standardized and handled via the `ego` package.*

**Project Structure**
```text
├── server/
│   └── rest/
│       ├── {public|service|...}/
│       │   ├── v1/
│       │   │   ├── api/
│       │   │   │   ├── {feature}/
│       │   │   │   │   ├── type.go
│       │   │   │   │   ├── {handler_name}.go
│       │   │   ├── container/
│       │   │   │   ├── wire_gen.go
│       │   │   │   ├── {feature}.go
│       │   │   │   └── ...
│       │   │   ├── router/
│       │   │   │   ├── {feature}.go
│       │   │   │   └── ...
│       │   │   └── routes.go
```

- **`api/`**: Contains multiple feature folders. Each feature folder houses the HTTP handlers relevant only to that feature. Features and their handlers must remain strictly isolated and independent from one another.
- **`router/`**: Manages HTTP routing. Every feature must specify its own router configuration here.
- **`routes.go`**: Acts as the central registry, aggregating and registering all routes from the individual feature routers.
- **Handlers**: Each `.go` file within a feature's API directory strictly manages a single endpoint (e.g., `generate_token_client.go`).
- **Error Handling**: You must use `c.ResponseError` to return errors within the API layer.
- **Tracing**: Span tracing must be initialized at the beginning of each handler.
- **Dependency Injection**: `wire_gen.go` inside the `container/` directory is automatically generated by Google's `wire` tool. Do not modify it manually.

- `generate_token_client.go`
```go
package access_token

import (
	"github.com/evenyosua18/auth-svc/app/entity"
	"github.com/evenyosua18/ego/code"
	"github.com/evenyosua18/ego/http"
	"github.com/evenyosua18/ego/tracer"
	"github.com/evenyosua18/ego/validator"
)

// GenerateTokenClient godoc
func (a *ApiToken) GenerateTokenClient(c http.Context) error {
	// tracer
	sp, ctx := a.tracer.StartSpanWithContext(c.Context(), "GenerateTokenClient", tracer.WithFilePath("api.access_token.generate_token_client"))
	defer sp.End()

	// get request
	var req entity.TokenGenerationClientRequest
	if err := c.RequestBody(&req); err != nil {
		return c.ResponseError(code.Wrap(err, code.BadRequestError))
	}

	// validate request
	if err := validator.Validate(req); err != nil {
		return c.ResponseError(err)
	}

	// call usecase
	res, err := a.accessTokenUc.GenerateFromOauthClient(ctx, req)
	if err != nil {
		return c.ResponseError(err)
	}

	return c.ResponseSuccess(res)
}
```

### 2. Usecase Layer
The Usecase Layer is responsible for encapsulating and handling business logic.
- This layer interacts with data and external systems by calling functions from the [Repository](#3-repository-layer) layer.
- This layer can call functions from other usecases to prevent code duplication and promote reusability.

**Project Structure**
```text
├── usecase/
│   ├── {feature}/
│   │   ├── type.go
│   │   ├── {usecase_name}.go
│   │   └── ...
```
- **Feature Folders**: Each folder represents a single domain feature and contains the usecases specifically related to that feature.
- **Single Responsibility**: Each usecase must be represented by a single `.go` file.
- **Entities**: All request and response structures must reside in the [Entity](#1-entity) layer.

**Sample Implementation**
- Directory Structure
```text
├── usecase/
│   ├── access_token/
│   │   ├── type.go
│   │   ├── validate_access_token.go
│   │   └── ...
```
- `type.go`

- `validate_access_token.go`
```go
package access_token

import (
	"context"
	"database/sql"
	"github.com/evenyosua18/auth-svc/app/entity"
	"github.com/evenyosua18/auth-svc/app/repository/db/access_tokens"
	"github.com/evenyosua18/ego/code"
	"github.com/evenyosua18/ego/tracer"
)

func (u *UsecaseAccessToken) ValidateAccessToken(ctx context.Context, request entity.ValidateAccessTokenRequest) (entity.ValidateAccessTokenResponse, error) {
	// tracer
	sp, ctx := u.tracer.StartSpanWithContext(ctx, "ValidateAccessToken", tracer.WithFilePath("usecase.token_validation.validate_access_token"))
	defer sp.End()

	// check if access token is revoked
	accessToken, err := u.accessTokenRepo.Get(ctx, access_tokens.FilterAccessToken{
		Id: sql.NullInt64{Valid: true, Int64: request.AccessTokenId},
	})
	if err != nil {
		return entity.ValidateAccessTokenResponse{}, sp.LogError(err)
	}

	return entity.ValidateAccessTokenResponse{
		UserId:         accessToken.UserId,
		ClientId:       accessToken.ClientId,
		ExpiredAt:      accessToken.ExpiredAt,
	}, nil
}
```

### 3. Repository Layer
The Repository Layer is responsible for data access logic, third-party API calls, and inter-service communication. Repositories must remain strictly independent of one another (a repository should not call another repository).

**Project Structure**
```text
 ├── repository/
 │   ├── db/
 │   │   ├── {table_name}/
 │   │   │   ├── type.go
 │   │   │   ├── {operation_name}.go
 │   │   │   └── ...
 │   ├── {microservice_name}/
 │   │   ├── {resource_name}/
 │   │   │   ├── {resource_name}.go
 │   │   │   ├── type.go
 │   │   │   ├── {operation_name}.go
 │   │   │   └── ...
 │   ├── {third_party_provider}/
 │   │   ├── {third_party_provider}.go
 │   │   ├── type.go
 │   │   ├── {operation_name}.go
 │   │   └── ...
 │   └── ...
```
- Each folder within `db` represents a specific database table (or the primary anchor table in joins). For example, if an `event` table joins with a `session` table, the repository remains in the `event` folder.
- Each folder within the `microservice` directory represents a specific domain resource (e.g., a `user` resource inside `authsvc`).
- Third-party providers and microservices must reside in their own dedicated folders, strictly separated from the `db` folder.
- Each table, third-party, and microservice folder must include a file named after the folder itself to register all its repository functions.
- Each repository folder must contain a `type.go` file defining its respective structs. For database folders, this file also holds raw SQL queries.
- When calling another microservice or third party via HTTP, use `request` function from `go-core` package.
- When calling another microservice, get the endpoint and auth key from config 
	- send the svc auth key via header with key `Authorization` with value `Bearer {svc auth key}`
	- the config format is `{microservice_name}_svc.base_url` and `{microservice_name}_svc.auth_key}`

**Sample Implementation**
- Directory Structure
```text
├── repository/
│   ├── db/
│   │   ├── access_tokens/
│   │   │   ├── type.go
│   │   │   ├── get.go
│   │   │   └── ...
```
- `type.go`
```go
package access_tokens

import (
	"context"
	"database/sql"
	"time"
	"github.com/evenyosua18/ego/db"
	"github.com/evenyosua18/ego/tracer"
)

type IAccessTokenRepository interface {
	Get(ctx context.Context, filter FilterAccessToken) (AccessToken, error)
}

type RepositoryAccessToken struct {
	db     db.IDBManager
	tracer tracer.Tracer
}

func NewAccessTokenRepository(db db.IDBManager, tracer tracer.Tracer) IAccessTokenRepository {
	return &RepositoryAccessToken{db: db, tracer: tracer}
}

// Struct and query logic
type AccessToken struct {
	Id         int64
	UserId     sql.NullInt64
	ClientId   string
	Scopes     string
	ExpiredAt  time.Time
	GrantType  string
	AccessType string
}

type FilterAccessToken struct {
	Id sql.NullInt64
}

func (f FilterAccessToken) SelectStatement() (string, []any) {
	query := `SELECT id, user_id, client_id, scopes, expired_at, grant_type, access_type FROM access_tokens WHERE 1=1`
	var args []any

	if f.Id.Valid {
		query += ` AND id = ?`
		args = append(args, f.Id.Int64)
	}

	return query, args
}
```
- `get.go`
```go
package access_tokens

import (
	"context"
	"github.com/evenyosua18/ego/logger"
	"github.com/evenyosua18/ego/tracer"
)

func (r *RepositoryAccessToken) Get(ctx context.Context, filter FilterAccessToken) (AccessToken, error) {
	// tracer
	sp, ctx := r.tracer.StartSpanWithContext(ctx, "Get", tracer.WithFilePath("db.access_tokens.get"))
	defer sp.End()

	// get executor
	db, err := r.db.GetExecutor(ctx)
	if err != nil {
		return AccessToken{}, sp.LogError(err)
	}

	// get query and args
	query, args := filter.SelectStatement()
	logger.DebugQuery(query, args)

	// query row
	row := db.QueryRow(query, args...)

	var t AccessToken
	err = row.Scan(&t.Id, &t.UserId, &t.ClientId, &t.Scopes, &t.ExpiredAt, &t.GrantType, &t.AccessType)
	if err != nil {
		return AccessToken{}, sp.LogError(err)
	}

	return t, nil
}
```

#### 3.1 Query Builder
DB query builder is stored in the `type.go` file inside the `repository/db` specific folder. This includes the request and response struct for the query builder.
The output for query builder is typically `query` with data type `string` and a list of `args` with data type `[]any`.

List of common DB operation files:
- `get.go`
- `list.go`
- `insert.go`
- `update.go`
- `delete.go`
- `count.go`

The primary purpose of the query builder is to encapsulate and reuse query logic across multiple repository operations.
For example, both `SELECT` and `UPDATE` statements often share the same `WHERE` clause conditions. Such shared logic should be extracted into a reusable function (e.g., `conditionQuery`) within the query builder.

**Structure Requirements for Query Builder**
- Should define the base query.
- Function for `select` operation should have a filter parameter and always add the `deleted_at IS NULL` condition.
- Function for `update` operation should have request and filter parameters.
- Function for `delete` operation should have a filter parameter.
- Function for `insert` operation should have a request parameter.
- Filter parameter is used to define the condition (`WHERE clause`) and should be separated into its own function to generate the condition query, so it can be reusable.
- Request parameter is used to define the value of the columns.
- Avoid creating redundant operations. For example, if you need a select query to get an event by ID and another to get an event by session ID, then the queries should be combined into one function that fetches by a filter parameter to make it more general, e.g., `GetEvent(ctx context.Context, filter FilterEvent) (Event, error)`.
- Avoid creating redundant functions in the query builder. For example, the condition query for the `WHERE` clause in `select` and `update` operations should be generated by the exact same function.

**Sample Implementation**
```go
package authorization_codes

// query builder

const (
	baseSelectQuery       = `SELECT id, code, code_challenge, code_challenge_method, redirect_uri, user_id, oauth_client_id, session_id FROM authorization_codes WHERE deleted_at IS NULL`
	baseInsertQuery       = `INSERT INTO authorization_codes(code, code_challenge, code_challenge_method, redirect_uri, user_id, oauth_client_id, session_id) `
	baseInsertValuesQuery = `VALUES (?, ?, ?, ?, ?, ?, ?)`
	baseUpdateQuery       = `UPDATE authorization_codes SET %s WHERE deleted_at IS NULL`
)

type (
	AuthorizationCode struct {
		Id                  int64          `db:"id"`
		Code                string         `db:"code"`
		CodeChallenge       string         `db:"code_challenge"`
		CodeChallengeMethod string         `db:"code_challenge_method"`
		RedirectUri         string         `db:"redirect_uri"`
		UserId              int64          `db:"user_id"`
		OauthClientId       int64          `db:"oauth_client_id"`
		SessionId           sql.NullString `db:"session_id"`
		IssuedAt            sql.NullTime   `db:"issued_at"`
		ExpiredAt           time.Time      `db:"expired_at"`
	}

	UpdateAuthorizationCode struct {
		IssuedAt sql.NullTime
	}

	FilterAuthorizationCode struct {
		Id           sql.NullInt64
		Code         sql.NullString
		NotIssuedYet bool

		pagination sqlhelper.Pagination
	}
)

func (a *AuthorizationCode) InsertQuery() (query string, args []any) {
	return baseInsertQuery + baseInsertValuesQuery, []any{
		a.Code, a.CodeChallenge, a.CodeChallengeMethod, a.RedirectUri, a.UserId, a.OauthClientId, a.SessionId,
	}
}

func (a *UpdateAuthorizationCode) UpdateQuery(filter FilterAuthorizationCode) (query string, args []any) {
	var updateClauses []string

	// issued at
	if a.IssuedAt.Valid {
		updateClauses = append(updateClauses, `issued_at = ?`)
		args = append(args, a.IssuedAt)
	}

	// get query condition
	conditionQuery, conditionArgs := filter.ConditionQuery()

	// set query
	query = fmt.Sprintf(baseUpdateQuery, strings.Join(updateClauses, ",")) + conditionQuery
	args = append(args, conditionArgs...)

	return
}

func (f *FilterAuthorizationCode) SelectQuery() (query string, args []any) {
	// get condition query
	query, args = f.ConditionQuery()

	// add base query
	query = baseSelectQuery + query

	// pagination
	query += f.pagination.Query()

	return
}

func (f *FilterAuthorizationCode) ConditionQuery() (query string, args []any) {
	// id
	if f.Id.Valid {
		query += ` AND id = ?`
		args = append(args, f.Id)
	}

	// code
	if f.Code.Valid {
		query += ` AND code = ?`
		args = append(args, f.Code)
	}

	// not issued yet
	if f.NotIssuedYet {
		query += ` AND issued_at IS NULL`
	}

	return
}
```

## Additional Layers and Modules

### 1. Entity
This folder contains structs representing the request and response payloads.
- Data transfers between Presentation and Usecase layers typically use Entity.
- File naming corresponds to the service features (e.g., an `event` service will have a corresponding `event.go` entity file)
- Entities transfer data across layers securely and explicitly.
- Request validation logic should typically be placed inside the entity folder (e.g., using validation tags or custom validation methods).

**Sample Implementation**
```text
├── entity/
│   ├── {usecase feature}.go
│   ├── access_token.go
│   └── ...
```
- `access_token.go`
```go
package entity

type TokenGenerationClientRequest struct {
	ClientId     string `json:"client_id" validate:"required"`
	ClientSecret string `json:"client_secret" validate:"required"`
	Scope        string `json:"scope"`
}

type TokenGenerationClientResponse struct {
	AccessToken string `json:"access_token"`
	ExpiresIn   int    `json:"expires_in"`
	TokenType   string `json:"token_type"`
}
```

### 2 Helper
The `helper` folder contains utility functions used across all layers, though you should default to the `ego` package utilities.

## 4. Development Patterns

### 4.1 General
- **Tracing**: Always start tracing span at the beginning of API, Usecase, and Repository functions using `tracer.StartSpanWithContext`. Remember to execute `defer sp.End()`. 
- **Context**: Pass `context.Context` from the HTTP request to the lower-level layers.
- **Resource Cleanup**: Always use `defer` for releasing resources such as db connections, file close, etc.
- **Nil Pointer Safety**: Validate references before usage.

### 4.2 Error Handling
- Use `c.ResponseError` inside API layer.
- Record errors to tracing using `sp.LogError(err)` in Usecase and Repository layers. All returned errors should wrap or use explicit app codes using `code` package.

### 4.3 Logging
- Use `logger` package from ego. Use `logger.DebugQuery` for SQL debugging inside repository layers.
